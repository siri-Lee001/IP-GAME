# IP-GAME 对话式 SOP

目标：用尽量少的来回，把一句话故事或短剧本做成可试玩互动影游，并明确当前交付是 `prototype` 还是 `final`。

核心原则：

- 先剧本，后角色，再节点图，再视频，最后网页。
- 只在信息不足时做一次补问，最多 8 个问题。
- 角色图来源、节点图来源、视频来源是强制确认点。
- 用户已有素材时，优先使用本地素材。
- 缺素材也可以继续，但必须说明降级级别，不能把保底结果说成最终质量。

## 第 1 步：故事输入

向用户要最小输入：

```text
给我一句话故事主题或短剧本就行。可以只写设定、冲突、角色、结局想法；信息不够的部分我会先自动推断，必要时下一轮一次性补问。
```

内部动作：

- 推断题材、情绪、视觉风格、互动深度、画面比例、长度。
- 写入 `story.meta.questionnaire`。
- 记录缺失字段，判断是否进入第 2 步补问。

## 第 2 步：一次性补问

只有信息不足时才问。可直接复制：

```text
我先根据你的故事拆出了一部分设定。为了继续生成互动影游，请你一次性补充这些项；不想填可以留空，我会用 AI 默认值。

1. 视频风格：电影级 CG 写实 / 国风水墨 / 二次元动画 / 3D 卡通 / 赛博科幻 / 手绘插画 / 其它
2. 故事类型：仙侠 / 悬疑 / 科幻 / 现代都市 / 治愈 / 热血 / 童话奇幻 / 其它
3. 整体情绪：轻松 / 温暖 / 紧张 / 热血 / 悲壮 / 其它
4. 故事长度：短 3-5 节点 / 中 6-10 节点 / 长 10-15 节点
5. 互动深度：单线 / 分支剧情 / 多结局
6. 画面比例：横版 16:9 / 竖版 9:16
7. 执行模式：逐步确认 / 全自动化
8. 其它补充说明或禁忌
```

如果必填项仍为空，只追问一次：

```text
这些关键项还不完整。是否允许我用 AI 默认值继续？回复“允许”我就继续；回复“不允许”我会等你补充。
```

字段落盘：

- 风格 -> `meta.questionnaire.visual_style`
- 类型 -> `meta.questionnaire.genre`
- 情绪 -> `meta.questionnaire.mood`
- 长度 -> `meta.questionnaire.length`
- 互动深度 -> `meta.questionnaire.interactivity`
- 比例 -> `meta.questionnaire.aspect_ratio`
- 执行模式 -> `meta.runMode`
- 补充说明 -> `meta.questionnaire.notes`

## 第 3 步：剧本与地图确认

在素材阶段前，必须先给用户确认结构化剧本。输出内容应包含：

- 故事梗概：一句话版和详细版。
- 主要角色小传：外貌、性格、动机、身份识别点。
- 服化道关键元素：服装、道具、标志物、禁用元素。
- 影游路线图：节点树、分支关系、结局路径。
- 节点清单：每个节点的场景、旁白/对白、选项、跳转、是否结局。
- 主题色建议：给 1-2 个备选。

结尾问法：

```text
请确认是否修改：
1. 角色小传
2. 分支路线/节点设置
3. 具体剧情/对白
4. 主题色方向

可以直接回复“确认，进入角色图阶段”，或回复“修改：……”。
```

确认后才能创建或更新 `story.json` 的正式节点结构。

## 第 4 步：角色图来源确认

角色一致性是硬门槛。没有确认角色图来源前，不进入节点图。

问法：

```text
下一步是生成/整理主要角色图，用来保证后续每个节点角色一致。请选择角色图来源：

1. 直接生成：我用角色定妆提示词生成多视角角色图。
2. 使用本地已有角色图：你提供三视图、设定图或参考图路径，我按这些素材作为身份锚点。
```

硬规则：

- 用户说已有角色图时，必须选本地素材路径，不能重新设计覆盖。
- 每个主要角色至少要有 `sheet` 或 `front`；有 `side`、`back`、`face` 更好。
- 角色存在重大状态变化时，拆成多个角色状态参考。
- 角色图确认后再运行或整理 `generate-character-prompts` 的结果。

本地素材补问：

```text
请把角色图放到项目目录的 assets/images/characters/，并告诉我每个角色对应的文件名；也可以直接给完整路径。
```

## 第 5 步：节点图来源确认

角色图确认后，进入图片资产阶段。除了每个节点的图片，还必须同步规划首页海报和路径总览图。每个项目至少需要：

- `sceneImage`：主场景图/海报式场景图。
- `storyboardImage`：视频参考合成故事板，左侧角色/道具身份参考区，右侧连续动作九宫格，用作视频生成参考。
- `posterPrompt` / 首页海报图：用于首页主视觉，不应直接等于随便一张节点截图。
- `mapOverviewPrompt` / 路径总览图：用于地图页居中放大展示，不应用流程图式 UI 临时拼接代替。

问法：

```text
下一步开始生成图片资产：每个剧情节点的主场景图、视频参考合成故事板（左侧角色身份条 + 右侧九宫格关键帧），以及首页海报和路径总览图。请选择图片来源：

1. 直接生成：我用节点提示词和总览图提示词生成图片。
2. 使用本地已有图片：你提供各节点对应图片，我负责路径映射与自检。
```

硬规则：

- 所有节点图必须和 `meta.questionnaire.aspect_ratio` 保持一致。
- 同一项目不能横竖混用。
- 节点图提示词必须继承角色参考图，避免换脸、换服装、换物种。
- 多角色故事板必须先做角色差异矩阵：逐个锁定脸型、发型轮廓、年龄感、体态、姿态语言、服装主色/材质和唯一道具；如果两个角色都是同题材高颜值人物，必须额外拉开成熟度、发际线、下颌轮廓、发型整洁度和动作气质。
- 首页海报必须是独立的主视觉 prompt，给标题与按钮留安全区。
- 路径总览图必须是完整的“高定游戏CG密集信息路径图”主视觉，先让玩家看懂路线，再配按钮跳转。
- 图片不得含可读文字、编号、水印、Logo。
- 结局节点应有 `endingImage`。

## 第 6 步：视频来源确认

节点图确认后，必须让用户选择视频来源：

```text
下一步生成节点视频，请选择视频来源：

1. 视频 API 生成：画质优先，需你确认可用接口和密钥；密钥、真实 endpoint、账号报文、签名 URL、模型私有参数都不会写进仓库。
2. 使用本地已有视频：你提供 mp4，我负责路径映射和自检。
3. 静态图合成视频：本地 ffmpeg 从图片合成 5-15 秒 mp4，适合测试和保底。
```

执行规则：

- 选 API：先确认 provider 能力、用户授权、密钥环境变量和项目本地配置，再调用；`providers/video.provider.json` 只提供通用能力模板，不包含任何真实服务商接口。
- 选本地视频：检查每个节点是否有 mp4，能否播放。
- 选静态合成：标记为 `prototype`，不要称为最终视频成片。
- API 失败可以降级到本地视频或静态合成，但要明确告诉用户。
- 视频生成完成后，除非用户要求逐条确认，否则直接继续生成成品网页、运行验证并交付。

建议：

- 快速试玩优先选静态合成。
- 成片质量优先选视频 API 或本地高质量视频。

### 视频 API 能力分级与质量策略

选 API 后，必须先给 provider 做能力分级，再决定提示词和参考图策略。不要默认所有 API 都能多参考图。

**A级：多参考图 / 角色参考图 + 分镜参考图**

- 适用：API 支持 `media[]`、多张 `reference_image`、角色图、物体图、首帧/参考视频等。
- 策略：每个主要角色单独传定妆图，同时传节点 `storyboardImage` 或 `sceneImage`。
- Prompt 必须显式绑定：`图1=陆沉舟，图2=云浅浅，图3=赵无咎，图4=本节点分镜参考`。
- 对每个角色写清独立身份锚点：脸型、发型、服装色、配饰、体型、姿态、道具；禁止互相换脸、融合脸、同款发型、同款衣服。
- 多角色同场时优先用中景/半身/全身，不要连续脸部大特写；关键近景只给当前说话/行动角色。
- 这是角色一致性最优策略，可标记为 `final` 的首选。

**B级：首尾帧 / 起止帧图生视频**

- 适用：API 支持 first frame + last frame，或首尾帧约束。
- 策略：为每个节点先生成“首帧”和“尾帧”两张最终画面，二者必须来自同一角色定妆锚点。
- 首尾帧里避免塞太多角色；同一镜头最多突出 1-2 个角色。
- Prompt 描述运动只发生在首尾帧之间，角色身份不得漂移。
- 若节点有复杂动作，拆成多个短 clip 后本地拼接，而不是让一个视频承载所有转折。

**C级：单参考图图生视频**

- 适用：API 只支持一张 `img_url` / 单参考图。
- 策略：不要直接把普通九宫格当唯一参考。先生成“视频参考合成图”：左侧是角色身份条（主要角色头像/全身/道具），右侧是本节点 1-3 个关键帧；全图无文字。
- Prompt 用位置绑定角色：`左侧第一位角色=陆沉舟，青白弟子服和打卡玉牌；左侧第二位=云浅浅，云白浅青衣裙和灵果`。
- 每个 clip 只聚焦 1 个主角色或 1 组关系；复杂节点拆 clip。
- 角色设计阶段必须拉开差异：发型轮廓、衣服主色、配饰、年龄感、气质动作。若角色都同为“白衣长发高颜值”，单参考图很容易同脸。

**D级：纯文生视频**

- 适用：API 只有文本 prompt，不能传图片。
- 质量边界：不能保证角色精准一致；若故事依赖多角色同脸区分，不能承诺 `final` 级角色一致性，只能作为 `prototype` 或风格预览。
- 策略：每个视频只放一个核心角色，减少多人同场和脸部近景；用强身份锚点写法：年龄、脸型、发型、衣服主色、唯一道具、动作习惯、镜头距离。
- 先让节点主场景图承担角色识别，视频只做氛围/动作补充；网页里保留节点海报和结局图增强连续性。
- 可用“静态图合成视频 + TTS/字幕/音效”保底，牺牲运动幅度，换取角色稳定。

**E级：本地静态合成或用户已有视频**

- 本地静态合成：角色最稳但运动最弱，只能标记 `prototype`，适合跑通交互。
- 用户已有视频：先检查分辨率、时长、可播放性，再映射路径；质量由素材决定。

通用质量门：

- API 额度紧张时，新任务默认 `duration <= 8s`；额度足够且用户同意时才增加到 10s 或更长。
- 禁止重复提交已成功或正在运行的任务；必须记录 task id 和状态。
- 先用一个节点做小样验证 provider 参数和画面策略；小样通过后再批量提交。
- 如果发现同脸、换脸、服装融合，不要盲目重刷；先判断是角色设计差异不足、参考图不合格、prompt 绑定不清，还是 provider 能力不足。
- 如果 API 报额度不足、权限不足、付费模式未开启或 provider 不支持所需能力，先停止新增任务并报告原因；不要为了测试反复消耗额度。
- 真实 API 视频只有在所有节点可播放、角色漂移风险可接受、验证通过时才标记 `final`。

## 最终包装：主题、网页、验证

生成 `game.html` 前，确认主题色：

```text
最后一步：网页主题色你希望用什么？可以给颜色名或色值，例如“墨绿”“深紫”“#6E56CF”。如果你不回答，我会按故事类型和情绪自动决定。
```

内部动作：

1. 写入或更新 `ui.json`。
2. 运行 `ip-game build-html <project-dir>`。
3. 运行 `ip-game verify <project-dir>`。
4. 若验证失败，列出问题和修复方案。
5. 交付 `game.html`、`story.json`、`ui.json`、`assets/`。

HTML 首页规则：

- 首页信息层默认展开，故事简介默认展开，帮助玩家先理解故事背景。
- 信息层必须是半透明磨砂质感，并提供整体“收起/展开封面信息”按钮，用户看完说明后可以手动收起查看海报。
- 首页海报仍是第一视觉主体，信息层不得做成厚重实心遮挡卡片。

## 输出物清单

最低可交付：

- `story.json`
- `ui.json`
- `game.html`
- `assets/prompts/`

原型试玩：

- 最低可交付内容
- `assets/images/`
- 首页海报图与路径总览图
- `assets/videos/` 中的静态合成视频
- `meta.deliveryTier = "prototype"`

最终交付：

- 原型试玩全部内容
- 角色参考图齐全
- 每个节点图比例一致且可读
- 每个节点视频可播放且质量达标
- `meta.deliveryTier = "final"`

## 不允许的行为

- 未确认角色图就生成节点图。
- 未确认视频来源就声称视频生成完成。
- 把静态图合成视频包装成高质量成片。
- 把密钥、Cookie、token 写入仓库。
- 在本地 file 模式下给素材路径追加缓存参数。
